In the previous single-shot design, AI would read `CVSS: AV:N + PR:N` and immediately pick

# SUB-TECHNIQUE DECISION

  Empty `subtechniques: []` IS ALWAYS VALID when:
    - Parent technique has no sub-techniques in MITRE ATT&CK (e.g. T1190, T1210)
    - CVE signals do NOT distinguish between sub-techniques of the chosen parent
    - Choosing an OS-specific or tool-specific sub-technique would be guess-work

  WHEN sub-techniques ARE appropriate (emit only if you have real signal):
    - A specific shell/interpreter is invoked post-exploitation → T1059.001/003/004/006/007
    - A specific OS-only primitive is the attack surface (e.g. Linux kernel → T1068)
    - A specific payload type is the delivery mechanism (.ps1, .jar, .war, ...)
    - A specific execution primitive is named (command injection → matching T1059.xxx)

  ANTI-HALLUCINATION GUARD:
    NEVER emit a sub-technique ID that is NOT in the current MITRE ATT&CK matrix.
    You will NOT be penalized for emitting `subtechniques: []` when no real signal exists.
    You WILL be penalized for emitting a fabricated ID like "T1190.001" or repeating
    the parent ID as its own sub-technique ("T1190" inside `subtechniques: ["T1190"]`).

  Format: when `subtechniques: []` is appropriate, write a `mapping_reason` that
  documents what you checked (e.g. "T1190 has no sub-techniques in MITRE; CVE
  describes HTTP endpoint RCE with no specific shell/interpreter invocation").

# ANTI-BIAS RULES (Phase 2)

- Do NOT use CVSS AV:N as a sole trigger to default to T1190. AV:N + client_side
  indicates an email/web delivery vector, NOT a server-side web exploit. Phase 1
  already disambiguated execution_surface.
- AVOID T1190 for client_side CVEs unless signals clearly indicate otherwise (deviation
  must be justified in `mapping_reasons`).
- AVOID T1204/T1566/T1189 for server_side CVEs even if entry_vector mentions "user"
  indirectly (e.g. "user input" in a web app form = T1190, not T1204).
- Honor the Phase 1 anchor block. If `execution_surface=client_side`, the typical
  territory is T1204/T1566/T1189; if server_side, T1190/T1210. Document deviations.

# INBOUND INTRUSION DISTINCTION

Use Phase 1 execution_surface as the FIRST filter. Then apply this 3-question test
only if execution_surface=server_side (otherwise the question is moot):

  1. What service / protocol is on the wire? (SMB / RDP / SSH / HTTP / DNS / SMTP)
  2. Vulnerability in transport/auth layer or application layer on top?
     - Transport/auth (compression bugs, channel handling, handshake flaws) → T1210
     - Application on HTTP (web framework, REST API, GraphQL) → T1190
     - VPN / remote-access gateway auth bypass → T1133
  3. Container escape / supply chain / CI/CD? → T1611 / T1195.xxx

# MEMORY-CORRUPTION PRIMITIVES (CWE-787, CWE-125, CWE-416, CWE-119, CWE-190)

When Phase 1 cwe_metadata.cwe_ids contains ANY memory-corruption CWE AND
execution_surface=server_side OR multi_hop, add these techniques IN ADDITION
to the initial-access primitive (T1190/T1210/T1204):

  1. T1203 (Exploitation for Client Execution) — Execution tactic
     The memory corruption itself IS the execution primitive. Heap/stack
     overflow, use-after-free, OOB write/read all map to T1203 because they
     are "exploitation for execution" — the attacker's goal is to execute
     attacker-controlled code via corrupted memory.

  2. T1499.004 (Endpoint DoS: Application or System Exploitation) — Impact
     When CVE description OR Phase 1 observable_side_effects mentions
     "process crash", "segfault", "service crash", "DoS", "denial of service":
     ADD T1499.004 to subtechniques and TA0040 to tactics. This is the most
     common observable outcome of memory-corruption exploits (stable RCE is
     rare due to ASLR/DEP).

  3. Evasive indicators for HTTP/web memory-corruption CVEs:
     - HTTP chunked transfer encoding (split oversized header across chunks)
     - URL/hex encoding of shellcode/ROP gadgets (%XX encoding)
     - Header obfuscation / smuggling (whitespace differences between WAF
       parser and target parser)
     - ROP chains, ASLR bypass, heap spraying (for actual memory corruption)

The kill chain for memory-corruption CVEs spans TA0001 → TA0002 → TA0040,
not just TA0001. T1190 alone is typically insufficient.
